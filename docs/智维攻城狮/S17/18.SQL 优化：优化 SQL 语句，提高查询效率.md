# SQL 优化：优化 SQL 语句，提高查询效率

## 引言

在现代应用中，数据库性能往往决定了系统的上限。无论是 OLTP 业务高并发写入，还是 OLAP 场景的大规模查询分析，SQL 优化都是工程师必须掌握的核心能力。
 本文从原理到实践，系统讲解 SQL 优化的关键方法，帮助你在实际项目中显著提升查询效率。



------

# 一、SQL 优化的核心思路

![img](https://imgoss.xgss.net/picgo-tx2026/QQ_1768274927210.png?tx)



SQL 优化的本质是：**减少扫描、减少计算、减少数据量、减少等待**。
 可以拆解为四个方向：

- **减少不必要的数据访问**（索引、过滤条件、覆盖索引）
- **减少不必要的计算**（函数、排序、分组优化）
- **减少不必要的行数**（分页、限制、提前过滤）
- **减少锁等待与资源竞争**（事务、隔离级别、并发控制）

------

# 二、索引优化：SQL 性能提升的第一抓手

## 2.1 建立合适的索引

常见适合建索引的字段：

- WHERE 条件中的字段
- JOIN 关联字段
- ORDER BY / GROUP BY 字段
- 高频查询的过滤字段

### 索引建立的黄金法则

- **区分度高的字段优先建索引**
- **组合索引遵循最左前缀原则**
- **避免在低选择性字段（如性别）建索引**
- **避免过多索引导致写入变慢**

------

## 2.2 覆盖索引（Covering Index）

覆盖索引能让查询只在索引中完成，不回表，性能极高。

示例：

```sql
SELECT id, name FROM user WHERE email = 'xx@example.com';
```

如果有索引 `(email, id, name)`，查询可直接在索引中完成。

------

## 2.3 避免索引失效

以下写法会导致索引失效：

| 写法                                     | 原因             |
| ---------------------------------------- | ---------------- |
| `WHERE DATE(create_time) = '2024-01-01'` | 对字段使用函数   |
| `WHERE age + 1 = 30`                     | 表达式计算       |
| `WHERE name LIKE '%abc'`                 | 前置模糊匹配     |
| `WHERE phone LIKE '138%'` ✔              | 后置模糊可用索引 |

优化方式：

- 使用范围查询：`create_time >= '2024-01-01' AND create_time < '2024-01-02'`
- 避免对字段做计算
- 尽量使用后缀模糊匹配

# 三、SQL 语句优化技巧

## 3.1 避免 SELECT *

`SELECT *` 会导致：

- 读取不必要的列
- 无法使用覆盖索引
- 增加网络传输量

应明确列名：

```sql
SELECT id, name, email FROM user;
```

------

## 3.2 合理使用 LIMIT

分页查询常见性能问题：

```sql
SELECT * FROM orders LIMIT 100000, 20;
```

OFFSET 越大，扫描越多。

优化方式：使用“延迟关联”：

```sql
SELECT o.*
FROM orders o
JOIN (
    SELECT id FROM orders ORDER BY id LIMIT 100000, 20
) t ON o.id = t.id;
```

------

## 3.3 避免在 WHERE 中使用 OR

OR 会导致索引失效：

```sql
WHERE a = 1 OR b = 2
```

优化方式：使用 UNION ALL：

```sql
SELECT ... FROM t WHERE a = 1
UNION ALL
SELECT ... FROM t WHERE b = 2;
```

------

## 3.4 JOIN 优化

JOIN 的核心原则：

- 小表驱动大表
- JOIN 字段必须有索引
- 避免跨类型 JOIN（如 int 与 varchar）

示例优化：

```sql
SELECT * FROM a JOIN b ON a.id = b.id;
```

确保 `a.id` 和 `b.id` 都有索引。

------

## 3.5 GROUP BY / ORDER BY 优化

排序和分组非常耗资源。

优化方式：

- 使用索引字段排序
- 避免对大表直接排序
- 使用覆盖索引减少回表
- 必要时使用临时表提前过滤

# 四、执行计划分析（EXPLAIN）

![img](https://imgoss.xgss.net/picgo-tx2026/QQ_1768275027343.png?tx)

SQL 优化必须依赖执行计划。

重点关注字段：

| 字段  | 说明                                                         |
| ----- | ------------------------------------------------------------ |
| type  | 访问类型（越靠前越好：system > const > ref > range > index > ALL） |
| key   | 使用的索引                                                   |
| rows  | 预计扫描行数                                                 |
| Extra | 是否使用临时表、排序等                                       |

示例：

```sql
EXPLAIN SELECT * FROM user WHERE email = 'xx@example.com';
```

如果 type = ALL，说明全表扫描，需要优化。

------

# 五、事务与锁优化

## 5.1 缩短事务时间

事务越长，锁持有时间越长，越容易阻塞。

建议：

- 只在必要的代码段开启事务
- 避免事务中执行耗时操作（如网络请求）

------

## 5.2 合理选择隔离级别

隔离级别越高，锁竞争越严重。

常见选择：

- OLTP：READ COMMITTED
- 金融类强一致：REPEATABLE READ 或 SERIALIZABLE

------

# 六、缓存与分库分表

当 SQL 优化到极限后，需要从架构层面优化。

## 6.1 使用缓存

- Redis 缓存热点数据
- 缓存查询结果
- 使用缓存过期策略避免雪崩

## 6.2 分库分表

适用于：

- 单表数据量 > 5000 万
- 高并发写入场景

常见方案：

- 水平分表（按用户 ID、时间等）
- 垂直拆分（按业务模块拆库）

------

# 七、SQL 优化实战案例（简版）

## 案例：慢查询 5 秒

原 SQL：

```sql
SELECT * FROM orders WHERE user_id = 1001 ORDER BY create_time DESC LIMIT 20;
```

问题：

- user_id 无索引
- ORDER BY create_time 需要排序

优化：

1. 建组合索引 `(user_id, create_time)`
2. 查询变为覆盖索引
3. 排序在索引中完成

优化后耗时：**5 秒 → 20 毫秒**

------

# 结语

SQL 优化不是单点技巧，而是一套系统方法： **索引设计 → SQL 写法 → 执行计划分析 → 架构优化**。
