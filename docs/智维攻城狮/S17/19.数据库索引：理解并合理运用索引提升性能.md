# 数据库索引：理解并合理运用索引提升性能

数据库性能优化中，**索引（Index）永远是最具性价比的手段之一**。无论是 MySQL、PostgreSQL 还是 MongoDB、Redis（其内部也有索引结构），索引的本质逻辑高度一致：**用空间换时间，让查询更快、更稳定、更可控**。

## 1. 索引的本质：为什么它能让查询变快？

索引的核心作用是减少数据库扫描的数据量。

- **没有索引**：全表扫描（Full Scan），逐行比较，时间复杂度接近 (O(n))
- **有索引**：基于 B+Tree / Hash / Bitmap 等结构快速定位，时间复杂度可降至 (O(\log n))

可以把索引理解为书籍的“目录”：

- 没有目录：从第一页翻到最后一页
- 有目录：直接跳到对应章节

数据库索引的目标就是**减少无效扫描**。

## 2. 常见索引类型与适用场景

### 🟦 2.1 B+Tree 索引（最常见）

![img](https://imgoss.xgss.net/picgo-tx2026/QQ_1768275217483.png?tx)

适用场景：

- 范围查询（BETWEEN、>、<）
- 排序（ORDER BY）
- 分组（GROUP BY）
- 等值查询（=）

特点：

- 有序
- 支持范围扫描
- MySQL、PostgreSQL 默认索引结构

### 🟩 2.2 Hash 索引

适用场景：

- 等值查询（=）
- 高并发 KV 场景（如 Redis）

特点：

- 不支持范围查询
- 不支持排序
- 查询速度极快

### 🟧 2.3 全文索引（Fulltext）

适用场景：

- 搜索引擎
- 文本匹配（文章、评论、日志）

### 🟨 2.4 Bitmap 索引（OLAP 常见）

适用场景：

- 低基数字段（性别、状态、布尔值）
- 大规模分析查询

## 3. 索引的核心原则：如何“合理”使用索引？

### 🟦 3.1 高选择性字段优先建立索引

选择性 = 不同值数量 / 总行数

- 高选择性（如 user_id、email） → 索引收益大
- 低选择性（如性别、状态） → 索引收益低甚至拖慢性能

### 🟦 3.2 频繁出现在 WHERE、JOIN、ORDER BY 的字段优先

索引的价值在于减少扫描，因此以下场景优先考虑：

- WHERE 条件字段
- JOIN 关联字段
- ORDER BY 排序字段
- GROUP BY 分组字段

### 🟦 3.3 复合索引遵循“最左前缀原则”

例如复合索引 `(a, b, c)`：

- 可用于：a、(a,b)、(a,b,c)
- 不可用于：b、c、(b,c)

**错误示例：**

```sql
WHERE b = 10 AND a = 5
```

虽然条件包含 a，但顺序不符合索引结构，可能无法完全利用索引。

### 🟦 3.4 避免索引失效的写法

以下写法会导致索引失效：

- 对字段使用函数
   `WHERE DATE(create_time) = '2024-01-01'`
- 隐式类型转换
   `WHERE id = '123'`（id 为 int）
- 模糊匹配前置通配符
   `WHERE name LIKE '%abc'`
- 计算表达式
   `WHERE age + 1 = 30`

正确写法应保持字段“原样”参与比较。

------

## 4. 索引的成本：不是所有字段都适合建索引

### 🟥 4.1 索引占用磁盘空间

每个索引都是一棵 B+Tree，数据越多，索引越大。

### 🟥 4.2 索引会降低写入性能

INSERT / UPDATE / DELETE 都需要维护索引结构。

- 写多读少 → 索引越少越好
- 读多写少 → 索引越多越好

### 🟥 4.3 过多索引会导致优化器选择错误

优化器可能选择“错误的索引”，反而降低性能。

## 5. 实战：如何为一张表设计合理索引？

假设有订单表：

```sql
orders(
  id,
  user_id,
  status,
  amount,
  create_time
)
```

### 目标 1：按用户查询订单

```sql
WHERE user_id = ?
```

→ 建索引：`INDEX(user_id)`

### 目标 2：按时间范围查询

```sql
WHERE create_time BETWEEN ... AND ...
```

→ 建索引：`INDEX(create_time)`

### 目标 3：用户 + 时间组合查询

```sql
WHERE user_id = ? AND create_time > ?
```

→ 建复合索引：`INDEX(user_id, create_time)`

### 目标 4：按状态筛选（低选择性）

status 只有 3 个值 → 不适合单独建索引
 但可作为复合索引的后置字段：

```
INDEX(user_id, status, create_time)
```

## 6. 如何验证索引是否生效？

### 使用 EXPLAIN 分析执行计划

重点关注：

| 字段  | 含义                                      |
| ----- | ----------------------------------------- |
| type  | 访问类型（越靠近 ref/const 越好）         |
| key   | 使用的索引                                |
| rows  | 扫描行数（越少越好）                      |
| Extra | 是否出现 Using filesort / Using temporary |

如果看到：

- **Using filesort** → 排序未走索引
- **Using temporary** → 分组未走索引
- **ALL** → 全表扫描

说明索引设计需要调整。

## 7. 索引优化的通用流程（可复用）

### 🟦 步骤 1：分析慢查询日志（slow query log）

找出耗时最高的 SQL。

### 🟦 步骤 2：使用 EXPLAIN 查看执行计划

判断是否走索引、扫描行数是否过大。

### 🟦 步骤 3：优化 SQL 写法

- 避免函数
- 避免隐式转换
- 避免前置通配符

### 🟦 步骤 4：调整索引结构

- 添加缺失索引
- 合并冗余索引
- 删除低效索引

### 🟦 步骤 5：持续监控

- QPS、TPS
- 索引命中率
- 磁盘占用

# 总结：索引是性能优化的第一生产力

合理的索引设计能带来：

- 查询速度提升数十倍甚至数百倍
- CPU、IO 压力显著下降
- 系统吞吐量提升
- 更稳定的响应时间

但索引不是越多越好，关键在于：

- **理解业务查询模式**
- **基于数据分布设计索引**
- **持续监控与验证**
